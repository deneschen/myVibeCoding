# SEI CERT C 安全编码聚合规范

版本：2026-07-05

来源目录：`secure-coding-standards/content/4.sei-cert-c-coding-standard`

适用范围：本文件只聚合 SEI CERT C Coding Standard 中与 C 语言直接相关的高质量规范内容，用于 C 代码安全设计、代码评审、静态分析配置、缺陷整改和 AI 生成代码约束。本文不是源站逐字镜像，不复制长篇示例、完整 analyzer 表格、附件或站点管理内容。

## 1. 使用定位

### CERT-C-AGG-001 来源和边界

本文覆盖 CERT C 源目录下的 front matter、rules、recommendations、back matter、admin 和顶层索引。源站站点工程、图片附件和构建配置不属于本文范围。

CERT C 的核心目标是减少会影响安全性、可靠性和可移植性的 C 语言缺陷。对本项目而言，本文是 C 代码参考规范；涉及 C++ 产品代码时，应使用
[sei-cert-cpp-secure-coding-aggregate.md](./sei-cert-cpp-secure-coding-aggregate.md)
和
[vehicle-embedded-linux-cpp-coding-standard.md](../vehicle-embedded-linux-cpp-coding-standard.md)。

### CERT-C-AGG-002 规则、建议和偏离

- 规则用于阻止明确缺陷。违反规则必须修复，或形成批准的偏离记录。
- 建议用于降低缺陷概率并提升可维护性。建议默认执行；不执行时应在评审中说明原因。
- 偏离不得以性能、便利或代码风格为主要理由。偏离必须证明不会引入缺陷或可利用漏洞。

偏离记录模板：

```text

```

### CERT-C-AGG-003 合规和工具门禁

合规检查应优先使用编译器告警、ISO/IEC TS 17961 相关静态分析、CERT C checker、动态分析和人工评审组合。工具不能稳定覆盖的规则必须进入人工 checklist。

最低门禁：

- 编译告警按项目策略作为错误处理。
- 启用 CERT C、bugprone、security、portability、undefined behavior 相关静态分析。
- 对内存、数组、字符串、整数、文件 I/O、POSIX、并发和信号相关代码执行边界测试。
- 对解析器、协议和外部输入处理代码执行模糊测试或等价压力测试。

## 2. 总体 C 安全原则

### CERT-C-GEN-001 不依赖未定义或实现定义行为

C 代码不得依赖未定义行为、未指定行为、实现定义行为、编译器扩展或平台副作用。必须特别关注整数溢出、无效移位、未初始化读取、越界访问、对象生命周期、严格别名、对齐、信号处理和库函数前置条件。

### CERT-C-GEN-002 所有输入默认不可信

来自文件、网络、IPC、环境变量、命令行、设备、共享内存、配置、持久化数据和外部库的数据必须验证长度、范围、编码、格式、状态机和资源消耗。

### CERT-C-GEN-003 资源必须有单一所有者

内存、文件、目录流、socket、锁、线程、临时文件、权限提升状态、环境修改和平台句柄必须有明确所有者和清理路径。多出口函数不得遗漏释放路径。

### CERT-C-GEN-004 错误必须显式处理

可失败函数的返回值、`errno`、短读短写、转换失败、分配失败、权限失败和资源耗尽必须检查。禁止吞错或把多种失败压缩成不可诊断的 `bool`。

## 3. 预处理和 API

### CERT-C-PRE-001 宏必须受限

宏不得产生多次求值、隐藏副作用、生成不透明控制流、污染命名空间或绕过类型检查。优先使用 `static inline` 函数、枚举常量、`const` 对象或局部封装。

### CERT-C-PRE-002 宏参数不得依赖副作用

传入函数式宏的实参不得包含自增、自减、赋值、函数调用副作用或依赖求值次数的表达式。宏定义必须对每个参数加括号，并清楚说明是否求值一次。

### CERT-C-API-001 API 契约必须清楚

公共 C API 必须说明所有权、缓冲区容量、空指针策略、错误返回、线程安全、阻塞行为、编码、时间单位和调用前置条件。不得要求调用方猜测隐藏约定。

## 4. 声明、初始化和作用域

### CERT-C-DCL-001 标识符不得冲突或保留

不得声明保留标识符、污染标准命名空间、创建不兼容声明、隐藏关键对象或依赖外部链接名冲突。头文件必须自包含并只暴露必要符号。

### CERT-C-DCL-002 对象必须初始化

自动存储期对象、结构体、数组、指针、锁和状态变量必须在使用前初始化。禁止读取未初始化内存、填充字节或未定义状态。

### CERT-C-DCL-003 作用域必须最小化

变量、函数和宏应限制在最小可见范围。文件内私有对象使用内部链接。避免全局可变状态；确需使用时必须说明并发、初始化和生命周期。

### CERT-C-DCL-004 结构跨信任边界必须防泄漏

结构体传出进程、网络、文件或信任边界前，必须初始化全部字段并清理填充字节，避免泄漏栈、堆或历史数据。

## 5. 表达式和控制流

### CERT-C-EXP-001 不依赖求值顺序

表达式不得依赖未指定求值顺序或副作用交错。复杂表达式应拆成简单语句，特别是函数实参、自增自减、赋值和宏实参。

### CERT-C-EXP-002 指针和对象访问必须有效

指针必须指向有效对象或允许的一尾后位置。不得解引用空指针、悬空指针、越界指针、未对齐指针或违反有效类型规则的对象表示。

### CERT-C-EXP-003 控制流必须显式

`if`、`for`、`while`、`switch` 等控制流应使用清晰代码块。`switch` 中每个 `case` 必须显式终止或标注 fallthrough。值返回函数所有路径必须返回值。

## 6. 整数和浮点

### CERT-C-INT-001 整数运算必须检查范围

加减乘除、取负、移位、类型转换、数组索引、对象大小、内存分配大小和格式化 I/O 中的整数值必须防止溢出、截断、符号变化和错误提升。

### CERT-C-INT-002 位操作必须使用无符号和明确宽度

移位和位运算优先使用无符号固定宽度类型。移位计数必须在范围内。跨协议、文件和网络边界必须明确字节序。

### CERT-C-FLP-001 浮点必须处理异常值

浮点代码必须定义 NaN、无穷大、舍入、异常状态和比较误差策略。不得把浮点直接用于权限、数组索引、状态机或安全边界判断，除非有明确范围验证。

## 7. 数组、字符和字符串

### CERT-C-ARR-001 数组边界必须可证明

数组索引、指针偏移、循环边界和对象大小必须来自同一已验证范围。多维数组、变长数组和跨函数缓冲区必须明确容量。

### CERT-C-STR-001 字符串必须有容量和终止策略

字符串复制、拼接、格式化和解析必须保证目标容量、空终止、编码和错误处理。禁止使用无法限制长度或无法诊断截断的接口。

### CERT-C-STR-002 格式字符串不得来自外部输入

日志、`printf` 族、`scanf` 族和自定义格式化接口不得把外部输入作为格式字符串。格式说明符必须匹配实参类型和宽度。

### CERT-C-STR-003 字符分类和编码必须安全

传给字符分类函数的值必须在有效范围内。多字节字符、locale、路径编码和网络编码必须显式处理。

## 8. 内存管理

### CERT-C-MEM-001 分配大小必须正确

动态分配前必须检查元素数量、元素大小、乘法溢出和对齐。不得把 `sizeof(pointer)` 误用为对象大小。

### CERT-C-MEM-002 释放必须匹配且唯一

每个动态对象必须释放一次且仅释放一次。禁止释放非堆对象、重复释放、释放后访问、跨 allocator 混用和错误生命周期转移。

### CERT-C-MEM-003 分配失败必须处理

`malloc`、`calloc`、`realloc`、平台分配器和资源创建失败必须有恢复、上报或安全降级路径。`realloc` 失败不得丢失原指针。

### CERT-C-MEM-004 敏感数据必须清理

密钥、token、密码、隐私数据和认证材料释放或复用前必须用不会被优化掉的方式清理。不得写入日志、core dump 或普通临时文件。

## 9. 文件、环境和平台边界

### CERT-C-FIO-001 文件 I/O 必须处理竞争和权限

文件打开、创建、重命名、删除、临时文件和目录遍历必须防止符号链接竞争、TOCTOU、宽权限、设备文件误用和路径规范化错误。

### CERT-C-FIO-002 I/O 结果必须完整检查

读写必须处理短读短写、EOF、错误、缓冲刷新、关闭失败和编码转换失败。安全相关写入应考虑原子更新和持久化语义。

### CERT-C-ENV-001 环境不可信

环境变量、当前目录、进程 locale、时区、umask、信号 mask 和运行时配置都可能被外部影响。安全决策不得依赖未经验证的环境。

### CERT-C-POS-001 POSIX 调用必须封装和校验

POSIX 代码必须处理 `errno`、`EINTR`、权限、资源耗尽、fd 泄漏、`CLOEXEC`、fork/exec 边界、线程取消、锁生命周期、条件变量、字节序和特权放弃结果。

### CERT-C-WIN-001 Windows 专有边界必须隔离

Windows 专有 API、路径、句柄、安全描述符和宽字符处理必须集中封装。跨平台代码不得散落条件编译和平台假设。

## 10. 错误处理、信号和并发

### CERT-C-ERR-001 错误状态必须可靠传播

错误处理路径必须保留根因、释放已获得资源，并避免覆盖关键 `errno` 或错误码。断言不能替代运行时错误处理。

### CERT-C-SIG-001 信号处理必须极小化

信号 handler 内只能调用异步信号安全操作，不得调用锁、日志、分配器、非重入库函数或复杂业务逻辑。复杂处理必须转交主循环或专用线程。

### CERT-C-CON-001 并发对象必须有明确同步

共享对象必须用 mutex、atomic、条件变量、消息队列或单线程所有权保护。不得通过 `volatile` 实现线程同步。

### CERT-C-CON-002 锁和等待必须有边界

不得在持锁期间执行阻塞 I/O、调用未知回调、等待外部事件或执行长耗时操作。多个锁必须有固定顺序。条件变量等待必须用谓词循环。

## 11. Misc 和可移植性

### CERT-C-MSC-001 禁用危险和废弃接口

禁用 `gets`、不受限字符串函数、不安全临时文件接口、危险随机数接口、无界格式化接口和已废弃库函数。替代接口也必须正确检查返回值。

### CERT-C-MSC-002 随机数必须匹配用途

安全用途必须使用密码学安全随机数源。普通伪随机数不得用于 token、nonce、密钥、认证、会话或安全抽样。

### CERT-C-MSC-003 代码必须可审计

安全关键 C 代码应避免过度宏、隐式全局状态、过深嵌套、重复逻辑和隐藏控制流。复杂逻辑应拆分为可测试的小函数。

## 12. C 代码评审清单

- 输入来源、信任边界、权限边界是否明确。
- 每个缓冲区是否有容量和终止策略。
- 每个整数表达式是否考虑溢出、截断、符号转换和边界值。
- 每个资源是否有单一所有者和失败路径清理。
- 每个可失败调用是否检查返回值和错误状态。
- 文件和路径操作是否防止符号链接、TOCTOU、设备文件和宽权限。
- POSIX、Windows、信号、线程和环境 API 是否集中封装。
- 并发访问是否有同步策略，锁顺序是否固定。
- 日志是否避免泄漏密钥、隐私和内部安全细节。
- 静态分析、动态分析、边界测试和失败路径测试是否覆盖本次修改。

## 13. AI 生成 C 代码规则

AI 生成或修改 C 代码时不得输出：

- 未检查返回值的 C/POSIX 调用。
- 未验证长度的数组、字符串和内存操作。
- `system`、`popen`、shell 拼接或不可信命令执行。
- 未封装的信号复杂处理、线程生命周期或特权切换。
- 全局可变状态、隐藏 singleton 状态或无界缓存。
- 未解释的宏魔法、C 风格多态或未定义行为依赖。

AI 必须同步说明相关 CERT C 规则族、修改范围、测试方式和未能验证的检查。
